# Mini-Drop 设计文档

> 一个分布式 Linux 性能采集与分析平台的复刻实现

## 一、架构图

```
                    │  ─ 鉴权/任务编排 │
                    │  ─ 状态机管理   │
                    │  ─ 心跳兜底监控 │

                  gRPC  │       │  HTTP (presigned URL)
                        ▼       ▼
                ┌───────────┐ ┌──────────┐
                │drop_server│ │  MinIO   │  对象存储
                │  (C++)    │ │（perf.data│
                │ 任务调度   │ │ + SVG）   │
                └─────┬─────┘ └────┬─────┘
                  gRPC│            │
                      ▼            │
                ┌───────────┐      │
                │drop_agent │──────┘ fork+exec mc cp 上传
                │  (C++)    │
                │ perf 采集 │
                └───────────┘
                      │
              (心跳同时主动 POST
               通知 apiserver)
                      ▼
                ┌───────────┐
                │PostgreSQL │◄────────┐
                └───────────┘         │
                      ▲               │
                      │ psycopg2      │
                ┌───────────┐         │
                │ analysis   │ poll_worker 每 5s 轮询
                │ (Python)   │ 待分析任务，触发分析
                └───────────┘─────────┘
```

### 1.2 四个组件职责

| 组件 | 语言 | 职责 |
|------|------|------|
| web_frontend | React 18 | 任务管理 UI、Agent 在线状态、火焰图与建议展示 |
| apiserver | Go + Gin + GORM | REST API、任务状态机持久化、gRPC 下发、对象存储签名 |
| drop（server+agent） | C++17 + gRPC | Agent 注册/心跳、任务调度队列、perf 采集执行 |
| analysis | Python 3 | perf.data 解析、火焰图生成、规则引擎、LLM 建议 |

四个组件通过三种协议互通：gRPC（C++ ↔ Go，双向流式心跳 + 单次任务下发）、HTTP REST（Go ↔ 前端，以及 C++ → Go 的内部通知）、psycopg2 直连（Python → PostgreSQL，绕开 apiserver 避免跨语言 RPC 开销）。

## 二、状态机迁移图

### 2.1 任务采集状态机（hotmethod_task.status）

```
PENDING(0) ──下发成功──▶ RUNNING(1) ──采集时长到──▶ UPLOADING(2) ──上传确认──▶ DONE(3)
    │                        │                           │
    │ 下发失败                │ 超时/Agent失败             │ 上传超时
    ▼                        ▼                           ▼
FAILED(4) ◄──────────────────┴───────────────────────────┘
```

每次迁移都同步写 `status_info`字段说明原因，例如：
- `"task created, waiting to dispatch"`
- `"dispatched to agent at 192.168.x.x"`
- `"collection finished, uploading to storage"`
- `"watcher timeout: no result within expected duration"`

**设计要点**：UPLOADING 状态不是 drop_agent 主动上报的，而是 apiserver 的 `startCollectionWatcher` 根据"采集时长是否已到但还未收到完成通知"这个时间窗口推断出来的。这是一个工程取舍（详见第四节），避免了修改 C++ 协议字段的成本。

### 2.2 分析状态机（hotmethod_task.analysis_status，独立维护）

```
PENDING(0) ──poll_worker拾取──▶ RUNNING(1) ──成功──▶ DONE(2)
    ▲                              │
    │                              │ 失败
    └──────────────────────────────┴────▶ FAILED(3) ──poll_worker重试──┐
                                                                        │
                                    └───────────────────────────────────┘
```

FAILED 会被 poll_worker 重新拾取重试，这是为了兼容"apiserver 在容器化环境下 exec Python 失败"的降级场景（详见第三节关键决策）。

### 2.3 Agent 在线/离线状态机（agent_info.online + 审计日志）

```
                    心跳（5s 一次）
                         │
                         ▼
            ┌─────────────────────────┐
   新注册 ──▶│        ONLINE           │
            └────────────┬────────────┘
                         │ 30s 无心跳
                         ▼
            ┌─────────────────────────┐
            │        OFFLINE          │──┐
            └─────────────────────────┘  │ 心跳恢复
                         ▲                │
                         └────────────────┘
```

每次 ONLINE→OFFLINE 或 OFFLINE→ONLINE 迁移都写入 `agent_audit_log` 表，前端 Agent 审计日志表格 10s 轮询展示。这条链路有**双重保险**：drop_server 的后台扫描线程（5s 周期，C++ 内存态）主动 POST 通知 apiserver；apiserver 自己也有一个 10s 周期的 Go goroutine 独立扫描 `last_seen` 字段兜底——即使 drop_server 进程崩溃，只要 Postgres 里的 `last_seen` 不再更新，Go 侧依然能在 30s 后判定离线。

## 三、关键决策

### 3.1 Analysis 触发机制：从"apiserver 直接 exec" 改为"轮询拾取"

**原始设计**（参照复刻指南）：apiserver 收到采集完成通知后，直接 `exec.Command("python3", "hotmethod_analyzer.py", ...)`调起分析脚本。

**问题**：容器化部署后，apiserver 运行在 Alpine 容器里，没有 Python 环境，无法跨容器 exec。

**决策**：改为 analysis 容器内常驻一个 `poll_worker.py`，每 5 秒查询 PostgreSQL，主动拾取 `status=DONE AND analysis_status IN (PENDING, FAILED)` 的任务并处理。apiserver 仍然保留原来的 `triggerAnalysis()` 调用（在非容器化的本机开发模式下依然有效），但它在容器内会失败并把 `analysis_status` 设为 FAILED——而这恰好被 poll_worker 的拾取条件覆盖，形成了一个意外但合理的降级路径。

**取舍**：轮询模式有 5 秒的延迟代价，但换来了开发模式和容器化部署模式的同一套代码兼容，不需要为两种环境写两套触发逻辑。

### 3.2 MinIO 预签名 URL：双 Client 而非字符串替换

**问题**：apiserver 容器内访问 MinIO 用的是 `minio:9000`（Docker 内部地址），但生成的预签名 URL 如果包含这个 host，浏览器无法解析。

**最初的错误方案**：生成内部地址的签名 URL 后，用字符串替换把 host 换成 `localhost:9000`。

**为什么错**：AWS SigV4 签名算法把 Host 头计算进了签名摘要，签名生成后换 host 字符串，会导致 MinIO 校验时签名和实际请求的 Host 不匹配，报 `SignatureDoesNotMatch`。

**最终决策**：维护两个 MinIO client——一个用内部地址（`minio:9000`）做上传/下载等容器间操作，另一个专门用浏览器可访问地址（`localhost:9000` 或宿主机 IP）初始化、只用来生成预签名 URL。这样签名时用的 host 和浏览器实际请求时的 host 完全一致。

### 3.3 perf 不打进镜像，运行时挂载宿主机二进制

**问题**：`linux-tools-generic` 装的 perf 版本绑定"构建镜像时的内核版本"，与"实际运行宿主机的内核版本"几乎不可能一致，跨机器部署必然失败。

**决策**：镜像内不装 perf，只装它依赖的用户态共享库（libdw、libunwind、libslang、libnuma、libtraceevent 等，这些库与内核版本无关）。运行时用 `HOST_PERF_PATH=$(readlink -f /usr/lib/linux-tools/$(uname -r)/perf)` 动态解析出宿主机真实的 perf 二进制路径，通过 docker-compose 的环境变量插值挂载进容器。

**为什么不用 `which perf`**：`/usr/bin/perf` 在 Ubuntu 上不是软链接，而是一段依赖 `lsb_release` 的检测脚本，容器内没有 `lsb_release` 会导致脚本本身报错。必须从 `/usr/lib/linux-tools/<内核版本>/perf` 这个真正的符号链接入口开始解析。

### 3.4 fork+exec 而非 system()

C++ 侧所有外部命令调用（perf、mc、curl）统一用 `fork() + execvp() + waitpid()`，不用 `system()`。原因：`system()` 经过 shell 解释执行，存在命令注入风险，且无法精确控制子进程生命周期（超时杀进程需要 `setpgid` + `killpg`，`system()` 做不到）。

## 四、取舍说明

| 决策点 | 选择 | 放弃的选项 | 取舍理由 |
|--------|------|------------|----------|
| 火焰图渲染 | analysis 直接产 SVG，前端 iframe 加载 | analysis 产 JSON 树，前端 d3-flame-graph 可交互渲染 | iframe 方案 5 行代码搞定，可交互渲染需要额外学习 D3，时间不允许 |
| 容器权限 | `privileged: true` | 精确枚举 `cap_add` 最小权限集合 | privileged 已验证可用，最小权限方案需要额外调试时间，演示截止前优先稳妥（`cap_add` 已写在配置里作为future work的切换点） |
| perf_event_paranoid | 写进 README 要求宿主机手动设置 | 容器内自动设置 | 这是内核级开关，容器命名空间无法修改宿主机的 `/proc/sys`，只能由人工在宿主机执行一次 |
| 上传方式 | fork+exec 调用 `mc cp` | 直接用 MinIO C++ SDK | 项目时间有限，mc 命令行工具比集成 C++ SDK 更快上手，且复刻指南本身就提到这是"COS 上传"的可选实现方式之一 |
| Agent 心跳判活 | 双重保险（C++ 主动通知 + Go 被动轮询） | 只用一套机制 | 单一机制在对方进程崩溃时会失效；双重保险增加约 60 行代码，换来了真正的容错能力 |

## 五、AI 协作章节

本项目全程使用 Claude Code 进行 AI 辅助开发，协作模式经历了三个阶段：

**阶段一（架构与骨架）**：在完全不了解项目背景的情况下，先让 AI 阅读复刻指南和题目要求，逐项拆解成模块清单和数据库表设计，再由我确认后开始写代码框架（proto 文件、Go 项目结构、7 张表）。

**阶段二（功能实现 + 调试）**：每天明确当天目标后，让 AI 编写对应模块代码，并在每次提交前要求列出"前置知识清单"，倒逼自己理解代码而不是直接复制。多次踩坑（CMake/gRPC 链接、GORM tag 缺失、SVG Content-Type、perf 跨机器可移植性）都是通过把真实报错信息贴给 AI，由 AI 定位根因后再决策修复方案，而不是由 AI 单方面决定要怎么改。

**阶段三（部署加固 + 文档）**：在我自己的虚拟机和docker容器之间反复验证 Docker 部署的可移植性问题时，AI 协助逐层排查（软链接→共享库→签名机制），每一层修复方案都先解释原理再由我确认是否执行，严格遵守"修改代码前必须询问"的工作规则。

**反思**：AI 协作显著加快了从"不会"到"能跑通"的速度，但也暴露出一个真实风险——如果只追求"让报错消失"而不去理解为什么会报错，知识留存会非常浅。本项目刻意在每个阶段要求 AI 补充前置知识、解释根因，正是为了避免变成"纯粘贴代码"。

## 六、性能自证小节

### 6.1 Docker 构建性能

引入 BuildKit 缓存挂载（`--mount=type=cache`）前后对比（同一台虚拟机，二次构建）：

| 阶段 | 优化前 | 优化后 |
|------|--------|--------|
| apt-get install（drop/analysis） | 每次重新下载，30~70 分钟（境外源）/ 2~5 分钟（国内源不缓存） | 缓存命中后 < 30 秒 |
| go mod download | 每次重新下载，约 10 分钟 | 缓存命中后 < 10 秒 |
| 完整重新部署 | 10~15 分钟 | 2~3 分钟（仅重新链接，不重新下载） |

### 6.2 采集链路实测时延

以 30 秒采样时长、99 Hz 频率的 CPU 采集任务为例：

```
任务下发 → Agent 拉取任务：< 5s（受心跳间隔限制）
perf record 执行：30s（采样时长本身）
mc cp 上传 perf.data：< 2s（本机 MinIO，文件通常 < 5MB）
poll_worker 拾取 + 分析：< 5s 轮询延迟 + 分析耗时（约 2~5s，含 perf script 解析）
端到端总耗时：约 45~50s（采样时长占了绝大部分，工程开销 < 15s）
```

### 6.3 心跳与离线检测时延

- 心跳间隔：5s（满足题目要求）
- 离线判定延迟：≤ 30s（C++ 侧主动检测，5s 扫描周期 + 30s 阈值）
- Go 侧兜底检测延迟：≤ 40s（10s 扫描周期 + 30s 阈值，作为 C++ 检测失效时的安全网）

## 七、如果再有 7 天我会做什么

按优先级排序：

1. **eBPF 采集器（最高优先级）**：用 bpftrace 实现一个 Block IO 延迟追踪探针，复用 analysis 容器已有的 `privileged: true` 权限，不需要改动 C++ Agent 代码，是当前架构下成本最低的实现路径。

2. **单元测试与集成测试**：当前覆盖率为 0，这是工程基线里最大的缺口。优先给 apiserver 的任务状态机（CreateTask/RetryTask/状态迁移）写 httptest 集成测试，覆盖正常路径 + 2 类异常路径（鉴权失败、任务不存在）。

3. **第二种用户态采集器**：py-spy 对 Python 进程采样，可以直接作用于 analysis 容器自身进程做演示，不需要额外目标进程，实现成本低于 eBPF。

4. **LLM Tool Use 改造**：把现有的直接 prompt 调用改成 Function Calling 模式，定义 `get_top_functions`、`get_rule_match`、`get_cpu_overhead` 三个工具，让 LLM 的结论可追溯到具体数据来源，而不是黑盒生成文字。

5. **Continuous Profiling**：常驻低频采样 + 时间轴回溯，需要新的后台采集调度逻辑和前端时间轴组件，工程量最大，优先级也最低。

6. **完善数据库索引与用户组/定时任务 API**：这两块属于"有更好，没有也不影响核心演示"的扩展项，留在最后。
